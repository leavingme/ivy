# 藤学（tengxue）— MVP 痛点评估

> 目标用户：女儿 Ivy（小学低年级）
> 评估日期：2026-06-16
> 原则：功能必须有产品形态 + 用户价值、边界明确、Local First（去 API Key 化）

---

## 痛点 1：识字 / 写字（字怎么写、组词、释义）

### 场景还原
- 写作业遇到生字（看到拼音忘了字形）
- 家长辅导时被问"这个字什么意思""怎么组词"
- 复习当天生字表

### 现有方案差距
| 现有 | 问题 |
|---|---|
| 百度/微信搜"X字" | 满屏广告、链接到不相关站点 |
| 新华字典 App | 内容权威但 UI 偏成人，小孩不爱看 |
| 家长口述 | 不随时可用；家长也不知道答案 |

### 差异化价值
- **字卡视觉化**：大字 + 部首 + 结构 + 拼音 + 动画笔顺一体（不是链接跳来跳去）
- **年级分级**：内置部编版 1-6 年级字表，按课内进度走
- **亲子共学**：组词卡片化，AI 出"造句挑战"（给词让孩子说句子）
- **收藏 / 错字本**：标记"我不熟"的字，自动汇总成复习清单

### 数据来源（Local First）
| 数据 | 来源 | License | 体积 |
|---|---|---|---|
| 字表 + 拼音 + 部首 | [pypinyin / unicode-data](https://pypi.org/project/pypinyin/) + [汉字字典表](https://github.com/elkmovie/character-table) | MIT | ~5MB |
| 部编版小学字表（1-6年级） | 公开教材大纲，手工整理成 JSON | 自有 | < 100KB |
| 释义 + 组词 | [现代汉语词典](https://github.com/pwxcoo/chinese-xinhua) (CC BY-SA) | CC BY-SA | ~10MB |
| 笔顺动画 | [hanzi-writer](https://hanziwriter.org/) + [hanzi-writer-data](https://www.npmjs.com/package/hanzi-writer-data) | MIT | 31.5MB |

### 技术方案
```
[字表 JSON] → 服务端按年级筛 → 客户端大字展示
              ↓
         [拼音 + 部首 + 结构] (静态数据)
              ↓
         [hanzi-writer 动画笔顺] (懒加载)
              ↓
         [组词 + 释义卡片] (静态 JSON + 卡片化 UI)
              ↓
         [错字本 localStorage] (无后端)
```

### MVP 范围建议
**v0.1 最小可用**：
- 首页：今日生字（5 个，按年级）
- 字详情页：大字 + 拼音 + 部首 + 笔顺动画 + 3 个组词 + 1 句释义
- 错字本（localStorage）
- ❌ 不做：AI 造句、复习曲线、家长后台

### 工作量估算
- 脚手架 + 首页 + 字详情页：1 个 commit
- 数据导入（部编版字表 + 释义 JSON）：1 个 commit
- 笔顺集成 + 错字本：1 个 commit
- **总计 ~3-4 个 commit，Claude Code 半天可跑完一版**

## 痛点 2：笔划顺序（笔顺动画 + 描红）

### 场景还原
- 老师要求"按笔顺写"，小孩写着写着就乱了
- 写字时家长反复纠正"先横后竖"
- 描红本买了很多，小孩不爱写

### 现有方案差距
| 现有 | 问题 |
|---|---|
| 百度搜"X字笔顺" | 结果杂乱、视频带广告 |
| 字典 App 内嵌笔顺 | UI 拥挤，笔顺粗略或收费 |
| 描红本 | 一次性的，浪费纸张 |

### 差异化价值
- **跟写模式（Tracing）**：hanzi-writer 内置 `quiz()` API，用户手指/鼠标按笔顺写，写错立即提示"应该从左到右"
- **重复练习**：错的笔画反复练，不限次数，比描红本灵活
- **进度奖励**：连续写对 10 个字解锁一个奖章（小马宝莉/汪汪队贴纸风格？需确认 Ivy 喜欢啥）

### 数据来源（与痛点 1 完全共用 hanzi-writer-data）
| 数据 | 来源 | License | 体积 |
|---|---|---|---|
| 笔顺 + 笔画 + SVG | hanzi-writer-data | MIT | 31.5MB |

### 技术方案
```
[字] → hanzi-writer.create()
        ↓
   animateCharacter()   模式 A：看动画
        ↓
   quiz()               模式 B：跟写测验
        ↓
   [实时反馈：正确 / 错 / 应是从 x,y 开始]
        ↓
   [累计积分 → 解锁奖章]
```

### MVP 范围建议
**v0.1 最小可用**：
- 单字页面：上方动画，下方跟写区
- 错笔画立即提示
- 跟写完成触发简单动画 + 鼓励语

### 工作量估算
- **与痛点 1 完全复用 hanzi-writer**：若先做痛点 1，痛点 2 只需再加 1 个 commit（quiz 模式 + 反馈 UI）
- 若先做痛点 2：脚手架 + hanzi-writer 集成 + 跟写页面 = 2 个 commit

### 关键陷阱
- **hanzi-writer-data 是按字懒加载**，不是一次性下 31.5MB——但首次访问单字会 fetch 一个 JSON，需要缓存
- 手写输入是 mouse/touch event，不是真正的"手写识别"，而是按笔画方向匹配——对低年级够用

## 痛点 3：看图写话（图 → 引导式作文）

### 场景还原
- 一年级语文作业："图上有谁？在做什么？心情怎么样？"
- 家长下班累，没精力引导
- 小孩坐在书桌前发呆 30 分钟写不出 50 字

### 现有方案差距
| 现有 | 问题 |
|---|---|
| 家长辅导 | 不随时可用、家长不知道"引导"方法 |
| 作文书 | 套路化、抄完不理解 |
| AI 通用助手（ChatGPT 等） | 不能固定图、不能按引导层级、可能给超纲答案 |

### 差异化价值
- **分步引导（关键差异化）**：
  1. 第一步：图上有谁/什么？（提取主体）
  2. 第二步：它们在做什么？（提取动作）
  3. 第三步：在哪里/什么时候？（环境）
  4. 第四步：心情怎么样？（情感）
  5. 第五步：把这些串成 50 字小段落
- **字数梯度**：3 句话（50 字）→ 5 句话（100 字）→ 8 句话（150 字），按年级切换
- **范文回评**：写完后给"好句子"+"可以更好"，不直接给范文让孩子抄

### 数据 / 算力
- **图像输入**：用户上传图 → 本地存储 / 对象存储
- **AI 算力**：本地的 Hermes 推理服务（`:8642/v1`），按用户偏好 Local First
- **引导模板**：写死在代码里的 JSON（不是 LLM 生成引导——稳定性 > 灵活性）

### 技术方案
```
[用户上传图] → 本地预览
        ↓
   [步骤 1：主体] → 引导："图上有谁/什么？" + 输入框
        ↓
   [步骤 2：动作] → 引导："它们在做什么？"
        ↓
   ...共 5 步
        ↓
   [本地拼接成段] → LLM 润色（可选） → 输出
        ↓
   [存到"我的作文"]（localStorage）
```

### MVP 范围建议
**v0.1 最小可用**：
- 上传一张图
- 5 步引导表单
- 拼接展示 + 一键复制
- ❌ 不做：图片理解（用户自己看图说话）、AI 润色、范文回评、家长后台

### 工作量估算
- 上传 + 表单分步：1 commit
- 拼接 + 收藏：1 commit
- AI 润色（v0.2 再加）：1 commit
- **总计 ~2-3 个 commit，但需要先决定是否引入 LLM 调用**

### 关键陷阱
- **看图写话的核心不是"看图"，是"写话"**——图像理解是 LLM 强项但本地小模型可能理解不准确
- v0.1 应该让小孩**自己看图说话**，AI 只做"拼接 + 检查字数"——更稳
- 图片存储涉及隐私（小孩照片），MVP 阶段不上传服务端，纯 localStorage 或 IndexedDB

## 综合对比

| 维度 | 痛点 1 识字 | 痛点 2 笔顺 | 痛点 3 看图写话 |
|---|---|---|---|
| 用户使用频次 | 高（每天写作业） | 中（写生字时） | 低（每周 1-2 篇） |
| 实现复杂度 | 中 | 低（高度复用 hanzi-writer） | 中-高 |
| Local First 纯度 | ★★★★★ 纯静态数据 | ★★★★★ 纯静态数据 | ★★★☆☆ 涉及 LLM（可选） |
| 数据准备成本 | 中（要整理部编版字表） | 零（hanzi-writer 已有） | 低（5 步表单写死） |
| 是否有产品形态 | ✅ 字卡，清晰 | ✅ 跟写界面 | ✅ 表单 + 作文页 |
| 失败风险 | 低 | 最低 | 中（AI 不可控时易出乱子） |

## 推荐顺序

### 🥇 先做痛点 1（识字 / 写字）
**理由**：
1. **频次最高**——每天写作业都用得上
2. **数据最完整**——hanzi-writer + CC BY-SA 词典都是现成的
3. **产品形态最稳**——字卡是确定的视觉结构，不会因为 AI 不可控变形
4. **为痛点 2 打基础**——字详情页天然包含笔顺模块，做完痛点 1 几乎免费拿到痛点 2

### 🥈 再做痛点 2（笔顺）
- 在痛点 1 的字详情页加跟写模式
- 复用 hanzi-writer，0 数据成本
- 加个奖章解锁增加粘性

### 🥉 最后做痛点 3（看图写话）
- 频次低、隐私敏感、AI 风险
- **建议先确认 Ivy 在哪个年级**，是否真有一二年级"看图写话"作业
- 若 Ivy 已经过了看图写话阶段，可以先做其他两个

## 给 Ivy 的"藤学 v0.1" 构想

按痛点 1 + 2 合并：

```
藤学 v0.1（识字 + 笔顺）
├── 首页 /              → 今日生字（5 个）
│                        + 收藏按钮（错字本）
├── /字/[code]          → 大字 + 拼音 + 部首 + 组词 + 笔顺动画 + 跟写测验
└── /错字本             → 收藏的字列表
```

技术栈：Next.js 16 + Tailwind v4 + hanzi-writer + Work Sans / Anthropic Sans + Canvas 流动曲线点阵背景（与 leavingme.cn 同源品牌）
